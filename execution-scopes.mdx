---
title: Execution Scopes
description: How full, partial, and single scopes control which nodes run when you call POST /api/v1/runs.
---

Every `POST /api/v1/runs` request includes a `scope` field that tells the orchestrator which nodes to execute. The default is `full`.

---

## Scope overview

<CardGroup cols={3}>
  <Card title="full" icon="play">
    Runs every node in the workflow graph in topological order. Use this for a fresh end-to-end run.
  </Card>
  <Card title="partial" icon="sliders">
    Runs a specific set of nodes plus their upstream dependencies. Useful for re-running one branch while reusing cached results from other branches.
  </Card>
  <Card title="single" icon="crosshairs">
    Same behavior as `partial` in the current orchestrator — expands to upstream dependencies of the target nodes. Use when you have all upstream outputs cached in `existingOutputs`.
  </Card>
</CardGroup>

---

## full

```json
{
  "workflowId": "clxwf...",
  "scope": "full",
  "inputValues": {
    "field_text_default": "A retro synthwave racing car at sunset"
  }
}
```

The credit hold is calculated by summing the base cost of every node in the workflow. See [Credits](/credits) for per-node costs.

---

## partial

```json
{
  "workflowId": "clxwf...",
  "scope": "partial",
  "nodeIds": ["gpt-image-node-1"],
  "inputValues": {},
  "existingOutputs": {
    "request-inputs": {
      "field_text_default": "Cached prompt from previous run"
    }
  }
}
```

`nodeIds` is required. The orchestrator automatically includes upstream dependencies of those nodes.

`existingOutputs` is a map of **node ID to output object**. Supply the outputs of nodes you already have results for — the orchestrator uses them as inputs without re-executing those nodes.

---

## Request-Inputs and inputValues

`inputValues` is merged into the `requestInputs` node at run start. Keys must match the `id` of a field in `node.data.fields`. The default scaffold uses `field_text_default`.

| Field type | Expected value |
| --- | --- |
| `text_field`, `text` | String |
| `number_field` | Number |
| `boolean_field` | Boolean |
| `image_field` (multi) | Array of image URL strings |
| `video_field`, `audio_field`, `file_field` | Single URL string |

---

## Concurrency guard

Only one run can be active per workflow at a time. A concurrent second request returns **409**:

```json
{
  "error": "A run is already in progress for this workflow",
  "runId": "clxrun..."
}
```

Poll `GET /api/v1/runs/{runId}` until status is terminal, then start your new run. You can also cancel the active run from the dashboard.
