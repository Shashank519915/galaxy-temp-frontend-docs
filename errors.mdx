---
title: Errors
description: HTTP status codes, error response shapes, and how to handle each error type from the public API.
---

All `/api/v1/*` routes return errors as JSON with at minimum an `error` string field. Validation failures include an additional `issues` array.

---

## Status codes

| Code | When it happens |
| --- | --- |
| **400** | Request body fails JSON parsing or Zod validation |
| **401** | Missing or invalid Bearer token |
| **404** | Resource not found, or found but belongs to another user |
| **409** | A run is already active for the target workflow |
| **429** | Unkey rate limit exceeded |
| **500** | Unhandled server error or insufficient credits at run start |

---

## 400 — Validation error

```json
{
  "error": "Invalid input",
  "issues": [
    {
      "code": "custom",
      "message": "Graph must include requestInputs and response nodes",
      "path": ["nodes"]
    }
  ]
}
```

Common messages from `lib/validation.ts`:

| Message | Cause |
| --- | --- |
| `"Refusing to persist an empty graph"` | `nodes` array was present but empty |
| `"Graph must include requestInputs and response nodes"` | Valid array but missing required node types |
| `"String must contain at least 1 character(s)"` | `name` field was empty |
| `"String must contain at most 120 character(s)"` | `name` exceeded 120 characters |

---

## 401 — Authentication error

The `Authorization` header is required and must use the `Bearer` scheme:

```http
Authorization: Bearer gx_your_key_here
```

Specific messages and their causes:

| Message | Cause |
| --- | --- |
| `"Missing Authorization header"` | No `Authorization` header in the request |
| `"Invalid API key"` | Key not found in local hash table (mock mode) |
| `"Invalid or unauthorized API key"` | Unkey rejected the key |
| `"API key is valid but has no identity/owner configured"` | Key verified by Unkey but has no Clerk user ID |

Verification order:
1. Check that `Authorization: Bearer <token>` is present.
2. If Unkey is configured and the key is not `gx_mock_*`, call Unkey's `keys.verifyKey`.
3. Otherwise, look up the SHA-256 hash in the `apiKey` table.

---

## 404 — Not found

```json
{ "error": "Workflow not found" }
```

The server returns 404 when a resource does not exist **or** when it exists but belongs to a different user — the two cases are not distinguished to avoid leaking information.

---

## 409 — Concurrent run conflict

```json
{
  "error": "A run is already in progress for this workflow",
  "runId": "clxrun..."
}
```

Poll `GET /api/v1/runs/{runId}` until status is terminal (`success`, `failed`, or `partial`), then start your new run. Or cancel the active run from the dashboard.

---

## 429 — Rate limited

```json
{ "error": "Rate limit exceeded" }
```

The `X-RateLimit-Reset` header tells you when the window resets. In local development with `gx_mock_` keys, the server simulates a 60 req/min limit — you will not see actual 429 errors in that mode.

---

## 500 — Server error / insufficient credits

Most 500s are unexpected errors. One business rule also uses this status code: insufficient credits at run start.

```
Insufficient credits. Estimated cost: 2.10M, but your balance is 0.50M.
```

Check `GET /api/v1/credits/balance` before starting runs on graphs with expensive nodes. For all other 500 errors, check the Vercel function logs or the Trigger.dev dashboard.

---

## Rate limit headers

Present on successful responses when Unkey returns limit metadata:

| Header | Meaning |
| --- | --- |
| `X-RateLimit-Limit` | Max requests in the current window |
| `X-RateLimit-Remaining` | Requests remaining before reset |
| `X-RateLimit-Reset` | Unix timestamp (seconds) when window resets |

---

## Webhook delivery failures

Webhook errors do not surface through the API. If a `node.completed` or `run.completed` event fails to reach your endpoint, the Trigger.dev task retries up to 3 times with exponential backoff. Check your Trigger.dev dashboard for delivery logs if events are missing.
