The Galaxy backend has two independent sets of routes:

<CardGroup cols={2}>
  <Card title="Public API /api/v1/*" icon="code" href="/quickstart">
    Bearer key auth. Documented in the API Reference tab. Use for integrations, scripts, and automation.
  </Card>
  <Card title="Dashboard routes /api/*" icon="browser">
    Clerk session cookie auth. Documented here. Power the canvas UI — only useful if you are building a custom frontend against the same backend.
  </Card>
</CardGroup>

<Warning>
Dashboard routes do not accept Bearer API keys. They require a valid Clerk session obtained by signing into the frontend. For programmatic access, use `/api/v1/*`.
</Warning>

## API key management

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/api/keys` | List keys for the signed-in user (values masked) |
| `POST` | `/api/keys` | Create a new key — raw token returned once |
| `DELETE` | `/api/keys/{id}` | Revoke a key by database ID |

Keys are stored as SHA-256 hashes. The full `gx_...` token is returned only in the `POST /api/keys` response body.

## Workflow management

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/api/workflows` | List all workflows including webhook fields |
| `POST` | `/api/workflows` | Create workflow with default scaffold |
| `GET` | `/api/workflows/{id}` | Full graph (nodes, edges, webhook config) |
| `PATCH` | `/api/workflows/{id}` | Update graph or metadata — same Zod rules as public `PUT` |
| `DELETE` | `/api/workflows/{id}` | Delete workflow and run history |
| `POST` | `/api/workflows/{id}/run` | Start a run from the dashboard |
| `POST` | `/api/workflows/{id}/cancel` | Cancel an in-flight run |
| `GET` | `/api/workflows/{id}/history` | Paginated run history with node-level results |
| `POST` | `/api/workflows/import` | Import a workflow from a JSON bundle |
| `GET` | `/api/workflows/{id}/export` | Export a workflow as a JSON bundle |
| `GET` | `/api/workflows/sample` | Fetch a pre-built sample workflow |

## Webhook configuration

There is no `/api/v1` route for webhook configuration — it is dashboard-only.

`POST /api/workflows/{id}/webhook`

```json
{ "webhookUrl": "https://your-server.com/galaxy-hooks" }
```

| Rule | Detail |
| --- | --- |
| URL format | Must start with `http://` or `https://` |
| Clear webhook | Send empty string to disable delivery |
| Secret on first save | `webhookSecret` generated as `whsec_<32 hex chars>` |
| Secret on update | Existing secret is preserved if only the URL changes |

The response includes both `webhookUrl` and `webhookSecret`. Store the secret securely — you need it to verify signatures. See [Webhooks](/webhooks) for the algorithm.

## Internal execution routes

Called by the canvas during and after run execution. Not intended for direct integration use.

| Method | Path | Description |
| --- | --- | --- |
| `GET` / `POST` | `/api/workflows/{id}/node-runs` | Incremental node run state updates |
| `POST` | `/api/workflows/{id}/node-runs/incremental` | Streaming updates during live execution |
| `POST` | `/api/workflows/{id}/node-runs/token` | Waitpoint token exchange for Trigger.dev |
| `POST` | `/api/workflows/{id}/runs/{runId}/reconcile` | Credit reconciliation after run completes |
| `POST` | `/api/upload` | File upload for canvas input fields |

---

## Single-node test routes

For testing individual AI providers without running a full DAG:

- `POST /api/execute/openrouter`
- `POST /api/execute/gemini`
- `POST /api/execute/crop-image`

These are useful for debugging provider connectivity. Auth requirements vary by route.

## Mapping to public v1

| Dashboard route | Public v1 equivalent |
| --- | --- |
| `POST /api/workflows` | `POST /api/v1/workflows` |
| `PATCH /api/workflows/{id}` | `PUT /api/v1/workflows/{id}` |
| `POST /api/workflows/{id}/run` | `POST /api/v1/runs` |
| `GET /api/credits/balance` | `GET /api/v1/credits/balance` |
