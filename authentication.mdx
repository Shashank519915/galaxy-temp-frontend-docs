Every request to `/api/v1/*` must include a Bearer token in the `Authorization` header.

<Tabs>
  <Tab title="cURL">
    ```bash
    curl "https://galaxy-temp-frontend.vercel.app/api/v1/workflows" \
      -H "Authorization: Bearer YOUR_API_KEY"
    ```
  </Tab>
  <Tab title="JavaScript">
    ```javascript
    const res = await fetch(
      "https://galaxy-temp-frontend.vercel.app/api/v1/workflows",
      { headers: { Authorization: `Bearer ${process.env.GALAXY_API_KEY}` } }
    );
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import os, requests

    res = requests.get(
        "https://galaxy-temp-frontend.vercel.app/api/v1/workflows",
        headers={"Authorization": f"Bearer {os.environ['GALAXY_API_KEY']}"},
    )
    ```
  </Tab>
</Tabs>

There are no query-string keys, session cookies, or other auth methods on the public API.

## Creating a key

<Steps>
  <Step title="Sign in">
    Open `https://galaxy-temp-frontend.vercel.app` and sign in with your account.
  </Step>
  <Step title="Open API and Outbound Webhooks">
    Click **API and Outbound Webhooks** in the left sidebar.
  </Step>
  <Step title="Create the key">
    Click **Create key**, give it a descriptive name (e.g. "Production"), and confirm.
  </Step>
  <Step title="Copy immediately">
    Copy the full `gx_...` token from the dialog. The raw token is shown **only once** — the server stores only its SHA-256 hash.
  </Step>
</Steps>

<Warning>
If you close the dialog without copying the key, you cannot retrieve it. Revoke it and create a new one.
</Warning>

For the programmatic equivalent, see [Dashboard API](/dashboard-api#api-keys-clerk).

## Key types

| Prefix | What it means | Verification |
| --- | --- | --- |
| `gx_...` | Production key via Unkey | `keys.verifyKey` call returns `identity.externalId` (your Clerk user ID) |
| `gx_mock_...` | Local / development key | SHA-256 hash matched against `apiKey` table in the database |

When `UNKEY_ROOT_KEY` and `UNKEY_API_ID` are not set in the environment, the server falls back to local hash verification for all key prefixes. In production on Vercel both variables are configured, so `gx_` keys go through Unkey.

## Managing keys

## Rate limiting

When requests are verified through Unkey, rate limit metadata is included in responses:

| Header | Description |
| --- | --- |
| `X-RateLimit-Limit` | Maximum requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining before the window resets |
| `X-RateLimit-Reset` | Unix timestamp (seconds) when the window resets |

Exceeded limits return **429**:

```json
{ "error": "Rate limit exceeded" }
```

In local development with `gx_mock_` keys, the server simulates 60 requests per minute with static header values.

## API playground

In the **API Reference** tab, click **Authorize** and enter your key. Depending on the UI, paste the full `Bearer gx_your_key_here` string, or just the key if the playground adds the prefix automatically.

Set the server selector to **Production** (the Vercel URL). Choosing local development only works when you have the backend running locally.

## Security practices

- Store keys in environment variables (`GALAXY_API_KEY`), never in committed code or screenshots.
- Use separate keys per environment or integration.
- Rotate a key immediately if it may have been exposed in logs, chat, or CI output.
- Delete keys you no longer use.
